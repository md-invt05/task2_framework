# Модульное приложение на C# / .NET 8

Практическая работа показывает модульную архитектуру, в которой ядро приложения не знает деталей конкретных модулей и работает только через контракт `IAppModule`.

## Что находится в решении

- `ModuleContracts` — контракты модулей и общие интерфейсы для демонстрации.
- `ModuleEngine` — ядро: загрузка DLL, каталог модулей, проверка зависимостей, топологическая сортировка, создание DI-контейнера.
- `Host` — консольное приложение, которое читает `appsettings.json`, загружает DLL из папки `modules`, инициализирует модули и запускает демонстрационные действия.
- `Plugins/*` — отдельные class library проекты модулей.
- `Tests` — xUnit-тесты на порядок запуска, ошибки и внедрение зависимостей.

## Модули

- `Core` — исходные данные для отчета и singleton-сессия приложения.
- `Logging` — консольное логирование и transient-метки операций.
- `Validation` — проверка данных отчета.
- `Export` — экспорт готового отчета в файл.
- `Report` — построение итогового отчета и запуск демонстрации.

## Почему модуль можно добавить без переписывания ядра

Ядро знает только контракт:

- имя модуля;
- список зависимостей;
- регистрацию сервисов;
- асинхронную инициализацию.

Новый модуль достаточно собрать в отдельную DLL, положить в папку `modules` и добавить имя в `appsettings.json`. Логика загрузки и порядка запуска уже универсальна и не зависит от конкретных модулей.

## Запуск

1. Собрать host и все модули:

```bash
dotnet build Host/Host.csproj
```

2. Запустить тесты:

```bash
dotnet test Tests/Tests.csproj
```

3. Запустить host-приложение:

```bash
dotnet run --project Host/Host.csproj
```

При сборке `Host` автоматически строит проекты модулей и копирует их DLL в папку `Host/bin/<Configuration>/net8.0/modules`.
